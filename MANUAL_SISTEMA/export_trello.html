Python Requests dateutil/ZoneInfo CSV Trello API v1

O que este código faz

Implementa um exportador completo de um quadro do Trello para múltiplos CSVs (boards, listas, membros, rótulos, cards, checklists, anexos e custom fields), criando uma subpasta exclusiva por execução com export_YYYYMMDD_HHMMSS (timezone America/Sao_Paulo). É plug-and-play para BI/ETL, com flatten consistente de entidades e referencial cruzado por IDs.

Inputs & dependências

  • Env vars obrigatórias: TRELLO_KEY, TRELLO_TOKEN (carregadas via dotenv do ambiente e do arquivo .env ao lado do script).
  • CLI: --board <BOARD_ID> (obrigatório), --out ./export_trello (opcional; default).
  • Libs: requests, python-dateutil, zoneinfo (Py≥3.9), python-dotenv.
# Exemplo de execução
python export_trello.py --board 5aBCdEF1234567890abcde12 --out ./export_trello

Painel de arquitetura (E2E)

  1. Bootstrap: carrega .env; valida TRELLO_KEY/TRELLO_TOKEN (senão aborta com SystemExit).
  2. Fetch consolidado (função fetch_board()):
    • /boards/{id} → metadados do quadro (name,url,desc,closed,dateLastActivity,prefs).
    • /boards/{id}/lists → listas ativas (name,closed,pos), e filtra apenas abertas.
    • /boards/{id}/members → (fullName,username).
    • /boards/{id}/labels → (name,color).
    • /boards/{id}/customFields → tolerante a erro (tenta; se 404/sem permissão, segue vazio).
    • /boards/{id}/cards → inclui attachments e checklists=all, com campos: name,desc,closed,url,idList,idMembers,due,start,dueComplete,labels,dateLastActivity,shortLink.
    • /boards/{id}/checklists → checklist e itens com metas de prazo/estado.
  3. Normalização:
    • flatten_cards() → expande rótulos, membros, anexos e mapeia idList → nome da lista; formata datas p/ ISO.
    • flatten_checklists() → uma linha por item de checklist, join por card_id.
  4. Persistência: escreve CSVs com csv.DictWriter (UTF-8, cabeçalho padronizado). Subpasta: --out/export_YYYYMMDD_HHMMSS.

Artefatos gerados (CSVs)

  • board.csv — 1 linha: id,name,url,desc,closed,dateLastActivity
  • lists.csv — apenas listas abertas: id,name,pos,closed,idBoard
  • members.csv — id,username,fullName
  • labels.csv — id,name,color
  • cards.csv — flatten de cards: card_id,card,list,labels,members,due,start,due_complete,url,short_link,closed,updated,desc,attachments_count,attachments_urls
  • checklists.csv — uma linha por item de checklist: checklist_id,checklist,card_id,item_id,item,state,pos,due,member_id
  • attachments.csv — anexos por card: card_id,card,attachment_id,name,url,bytes,mimeType,date,isUpload
  • custom_fields.csv — se existirem: id,name,type,idModel,modelType,pos,options

Datas são normalizadas para ISO via dateutil.parser quando possível.

Estratégia de saída & naming

  • Diretório raiz: parâmetro --out (default ./export_trello).
  • Subpasta por execução (não sobrescreve): export_YYYYMMDD_HHMMSS em fuso America/Sao_Paulo.
  • Idempotência operacional por run → cada execução gera um snapshot completo do board.

Gestão de falhas & hardening

  • Erros de credencial: ausência de TRELLO_KEY/TRELLO_TOKEN interrompe com SystemExit.
  • HTTP != 200: trello_get() aborta com mensagem detalhando status_code e corpo.
  • Campos customizados: se o endpoint não existir/sem permissão, o processo continua (arquivo omitido).
  • Checklist/Attachments: sempre normalizados; URLs de anexos consolidados em cards.csv e detalhados em attachments.csv.

Deliverables & time-to-value

  • Full-fidelity export para consumo por Data Studio/Power BI/ETLs.
  • Chaves de join: card_id e IDs de checklist/attachments para relacionar fatos/dimensões.
  • Taxa de adoção: mínima fricção operacional; basta o Board ID e credenciais.

Output de execução (STDOUT): Export OK: <outdir> (Xs).
