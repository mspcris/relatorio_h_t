Não está em Produção

Um serviço FastAPI para administração de usuários em arquivo JSON, com endpoints sob /admin/users. Tenta expor o router do .htpasswd em /users-api, mas isso é invalidado por um bug.

O que faz

  • Persistência: users.json em /var/lib/users_api/users.json.
  • Escrita atômica: grava em users.json.tmp e executa os.replace.
  • Controle de concorrência: threading.Lock em memória do processo.
  • Modelos: User e NewUser (Pydantic). EmailStr opcional.
  • Router externo: importa admin_htpasswd.router como users_router.

Endpoints efetivos

Devido ao bug de re-inicialização do app, apenas os endpoints abaixo ficam ativos.

GET
/admin/users → lista List[User].
POST
/admin/users → cria usuário; valida duplicidade por username; retorna {"ok": true, "id": …}.
POST
/admin/users/{id}/disable → marca disabled conforme payload {"disabled": true|false}.
POST
/admin/users/{id}/reset_password → stub; exige "password" no payload e não persiste senha.
DELETE
/admin/users/{id} → remove pelo id.

Bugs e riscos

  • App sobrescrito: cria app = FastAPI(...), inclui users_router e depois recria app = FastAPI(...). O router em /users-api é descartado.
  • Import duplicado: from fastapi import FastAPI aparece duas vezes.
  • Senha não persistida: NewUser.password é ignorado; reset_password não grava em lugar algum.
  • Sem auth: endpoints administrativos sem autenticação/autorização.
  • Concorrência limitada: threading.Lock protege apenas dentro do mesmo processo; múltiplos workers não sincronizam.
  • Pasta não garantida: não cria dirname(DATA_PATH) antes de salvar.
  • Tipos soltos nos POSTs: disable_user e reset_pw recebem dict cru, sem validação Pydantic.
